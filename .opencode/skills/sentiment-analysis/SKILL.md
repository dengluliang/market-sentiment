# 舆情分析 Skill

## 概述

本 Skill 提供完整的全网舆情分析流水线：从用户需求解析、多平台数据采集、情感分析到飞书文档输出。

## 环境依赖

- **MediaCrawler**: 项目内 `MediaCrawler/` 目录
- **Python**: 3.11+
- **uv**: Python 包管理器（`~/.local/bin/uv` 或系统 PATH 中）
- **飞书 MCP**: 需在 `opencode.json` 中配置飞书 MCP URL
- **KOL/媒体库**: 同目录下 `kol_media_list.json`
- **平台配置参考**: 同目录下 `platform_config.md`

## 工作流程（7步）

### Step 1: 解析需求

从用户消息中提取：
1. **分析目标**：品牌/产品/事件名称
2. **关键词拆解**：将目标智能拆解为多个搜索关键词
   - 示例："小米17发热卡顿" → `"小米17发热,小米17卡顿,小米17 发热,小米17 卡顿"`
   - 示例："华为Mate70 口碑" → `"华为Mate70,Mate70体验,Mate70评价,华为Mate70怎么样"`
   - 规则：主关键词 + 同义/近义变体 + 带空格版本
3. **时间范围**：**必须询问用户**想爬多久的数据（最近3天/一周/两周/一个月）
4. **确认**：列出拆解的关键词和时间范围，等用户确认后再执行

### Step 2: 配置爬虫

修改项目内 `MediaCrawler/config/base_config.py`：

```python
KEYWORDS = "关键词1,关键词2,关键词3"  # Step 1 确认的关键词
CRAWLER_MAX_NOTES_COUNT = 200         # 每平台最大帖子数
CRAWLER_MAX_COMMENTS_COUNT_SINGLENOTES = 50  # 每帖最大评论数
ENABLE_GET_COMMENTS = True
SAVE_DATA_OPTION = "csv"
ENABLE_CDP_MODE = False               # 使用 Playwright 模式
```

同时配置各平台的时间筛选参数（参考 `platform_config.md`）。

### Step 2.5: 数据隔离（防止多次分析数据互相干扰）

每次新的舆情分析任务执行前，**必须**进行数据隔离处理：

1. **检查是否存在旧数据**：检查项目内 `MediaCrawler/data/` 下各平台目录是否有当天的 CSV 文件
2. **如果存在旧数据**：将整个 `data/` 目录重命名归档为 `data_archive/{YYYY-MM-DD}_{关键词简写}_{时间戳}/`，然后创建新的空 `data/` 目录
3. **归档命令**：
   ```bash
   export PATH="$HOME/.local/bin:$PATH"
   cd MediaCrawler
   ARCHIVE_NAME="data_archive/$(date +%Y-%m-%d)_${KEYWORD_SHORT}_$(date +%H%M%S)"
   mkdir -p "$ARCHIVE_NAME"
   # 只移动有内容的平台目录
   for platform in weibo xhs douyin bili zhihu; do
     if [ -d "data/$platform" ] && [ "$(ls -A data/$platform 2>/dev/null)" ]; then
       mv "data/$platform" "$ARCHIVE_NAME/"
     fi
   done
   ```
4. **确保干净环境**：归档后确认 `data/` 下无残留 CSV 文件再开始爬取

**规则**：
- 同一个会话内多次运行同一任务（如重试失败的平台）不需要归档
- 不同关键词/不同任务之间必须归档隔离
- 归档目录命名示例：`data_archive/2026-05-19_小米17发热_134530/`

### Step 3: 依次爬取5个平台

执行顺序：**微博 → 小红书 → 抖音 → B站 → 知乎**

每个平台的执行命令：
```bash
export PATH="$HOME/.local/bin:$PATH"
cd MediaCrawler
uv run main.py --platform {platform} --lt qrcode --type search
```

平台代码：`wb` / `xhs` / `dy` / `bili` / `zhihu`

**异常处理**：
- 登录态过期（弹出扫码页面超时）→ **暂停并提醒用户扫码**
- 浏览器崩溃（TargetClosedError）→ 记录已采集数据，继续下一个平台
- 无搜索结果 → 在报告中注明"该平台暂无相关讨论"

每个平台完成后向用户报告进度和采集量。

### Step 4: 数据汇总

1. 统计各平台 CSV 文件行数：
   ```bash
   find MediaCrawler/data -name "*.csv" -exec wc -l {} \;
   ```
2. 汇报采集总量

### Step 5: 情感分析 + 观点提取

1. **读取数据**：分批读取各平台 CSV 评论文件（每批约 300-500 条）
2. **情感标注**：对每批评论进行正面/负面/中性分类
3. **观点聚类**：提取高频观点和核心论点
4. **KOL匹配**：将评论/帖子的 `nickname` 与 `kol_media_list.json` 中的账号名匹配
   - 命中的内容归入"媒体与KOL报道"章节
   - 记录其互动数据（点赞、评论数等）
5. **竞品识别**：从评论中自动识别提到的竞品品牌
6. **风险评估**：基于负面占比、情绪烈度、传播趋势进行风险定级

### Step 6: 生成飞书文档

使用 `feishu_create-doc` 工具，按以下模板输出到飞书个人知识库。

**文档命名规则**：`「{分析目标}」舆情分析报告`
- 示例：`「小米17发热卡顿」舆情分析报告`
- 示例：`「华为Mate70口碑」舆情分析报告`

**wiki_space**: `my_library`

### Step 7: 收尾

向用户提供：
1. 飞书文档链接
2. CSV 原始数据保存路径（`MediaCrawler/data/{platform}/csv/`）

## 报告模板结构（十章）

根据实际数据量灵活调整章节详略。数据充足时十章全输出，数据不足时精简或合并。

### 文档开头

```
报告类型：{产品口碑分析 / 热点事件舆情分析 / 品牌监测报告}
监测周期：{起始日期} — {结束日期}
报告生成日期：{今天日期}
报告密级：内部参考
---
```

### 一、核心事件概览（必须输出）

- 事件/产品背景描述（2-3段）
- 核心数据（采集量、平台覆盖、情感分布概览）
- 核心问题/亮点摘要（3-5条 bullet points）

### 二、产品/事件核心信息一览（有对比数据时输出）

- 用 lark-table 展示产品参数对比、版本对比、或事件时间线
- 如无明确对比维度可省略本章

### 三、销量/市场影响信号（有信号时输出）

- 直接影响信号（退货、品牌迁移言论等）
- 间接市场信号（竞品引流、潜在用户劝退等）
- 如无明确市场影响可省略本章

### 四、全网口碑分析（必须输出）

- **情感分布表**（lark-table：倾向/占比/核心表达）
- **负面聚焦点**（编号列表，8-10条）
- **正面/中性聚焦点**（编号列表，5条）

### 五、媒体与KOL报道（匹配到KOL库时输出）

- 分为"官方回应"和"KOL观点"两部分
- 每条引用格式：
  ```
  **@{账号名}**（{身份标签} / {平台} / {粉丝量或互动数据}）
  - 核心观点："{直接引用}"
  - 互动数据：{点赞/转发/评论数}
  ```
- 如未匹配到任何 KOL，本章精简为"本次监测期内未发现头部KOL专题报道"

### 六、核心社交媒体反馈（必须输出）

按平台分节，每个平台包含：
1. **平台概况**（1-2句：采集量+负面占比+讨论特点）
2. **核心负面观点**（3-5条，含用户昵称/地区/互动数据）
3. **核心正面/中性观点**（2-3条）
4. **平台舆情特征**（1句总结）

最后附 **社交媒体综合评估表**（lark-table：平台/负面占比/核心负面关键词/核心正面关键词/影响力）

### 七、竞品动态与竞争格局（有竞品对比数据时输出）

- 竞品在评论区的活跃表现（lark-table）
- 竞品借势风险评估
- 如评论中未出现明显竞品讨论可省略本章

### 八、舆情风险评估与预警（必须输出）

- **风险矩阵表**（lark-table：风险项/等级/说明），等级分为：高/中高/中/中低/低
- **潜在爆点**（编号列表，3-5条）

### 九、策略建议（必须输出）

分层建议（每层3-5条）：
1. **舆情应对建议**（紧急，本周内）
2. **产品/技术建议**（如适用）
3. **传播策略建议**（两周内）
4. **中长期品牌建议**

每层用 callout 块标注时间紧迫度。

### 十、总结（必须输出）

三段式：
1. 当前状态判断（一句话定性）
2. 核心矛盾/关键问题（一段分析）
3. 窗口期与展望（一段建议）

### 文档结尾

```
---
数据来源：微博、小红书、抖音、B站、知乎等公开社交媒体平台。
数据采集工具：MediaCrawler
分析模型：Claude
采集周期：{起始日期} — {结束日期}
```

## KOL 匹配规则

1. 读取同目录下 `kol_media_list.json`
2. 在采集的帖子/评论数据中，将 `nickname` 字段与库中各平台的账号名进行**模糊匹配**（包含即命中）
3. 命中的内容提取到"媒体与KOL报道"章节
4. 拉黑名单中的账号即使匹配也不放入报告

## 注意事项

1. **每次执行前必须和用户确认关键词拆解和时间范围**，不要自行假设
2. **数据隔离**：每次新任务开始前必须执行 Step 2.5 的归档操作，避免不同分析任务的数据混在一起。这是最容易遗忘但最关键的步骤
3. **数据采集需要时间**（30-60分钟），过程中保持向用户报告进度
4. **登录态过期**是最常见的中断原因，检测到时立即暂停提醒用户
5. **飞书文档大小有限**，如评论数据量极大（>5000条），分析时采用分层抽样策略
6. **报告中的典型评论引用**应优先选择互动数据高（点赞多）的内容
7. **所有 lark-table 的 lark-td 内容前后必须有空行**，这是飞书 Markdown 的语法要求
8. **建议每次新的舆情分析任务在新会话中执行**，保持上下文干净，能分析更多数据
